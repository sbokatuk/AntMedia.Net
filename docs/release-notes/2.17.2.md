## What's changed

First working release. The packages previously on nuget.org were `1.0.0.1-beta1` placeholders —
scaffolding with no usable binding in them — so treat this as the initial release rather than an
upgrade. Everything below is new.

Android binds [WebRTC-Android-SDK][android-sdk] at tag `v2.17.2`, built from source. iOS binds
[WebRTC-iOS-SDK][ios-sdk] at commit [`47e2b02`][ios-commit], dated 2025-11-14, which is the tip of
`master`.

**The package version follows the Android SDK, not the iOS one**, because the iOS SDK does not
have a version to follow. Upstream publishes none for it: the repository has one tag (`v2.10.0`,
2020), no GitHub releases, no CocoaPods pod, `MARKETING_VERSION = 1.0` in the Xcode project, and
Ant Media's documentation is versioned by *server* release rather than by SDK. A commit is
therefore the only precise way to say which iOS code you are getting, and `2.17.2` should be read
as "the Android SDK line, plus iOS master as of the date above" rather than as a claim that both
sides are at 2.17.2.

## ⚠️ Read this before writing iOS code

**Android and iOS do not expose the same API**, and the metapackage does not hide that. It gives
you the right binding for whichever target framework you build; it does not give you one API
across both. The sample shows the shape that works in practice — a small interface per platform
behind shared UI. See [`samples/`](../../samples).

The reason is that **there is nothing usable to bind on iOS**. WebRTCiOSSDK is written in Swift and
exposes almost nothing to Objective-C: its generated header declares exactly two `@objc` classes,
`AntMediaClient` (an `init` and two delegate callbacks) and `Config` (an `init`). `setOptions`,
`publish`, `play`, `AntMediaClientDelegate`, `AntMediaClientMode` — all Swift-only, and therefore
invisible to a .NET binding.

So `AntMedia.Net.iOS` binds an **`@objc` facade this repository maintains and compiles into the
framework**, not the SDK's own surface:

| Type | What it is |
| --- | --- |
| `AMSClient` | the client — `SetWebSocketServerUrl`, `InitPeerConnection`, `Publish`, `Play`, `Stop`, `SetLocalView`, `SetRemoteView`, camera/mic/zoom control, data channel |
| `AMSClientDelegate` | 20 optional callbacks — connect, publish/play started and finished, track added and removed, data received, errors |
| `AMSMode` | `Join`, `Play`, `Publish`, `Conference`, `Unspecified` |
| `AMSStreamInformation` | a rendition reported by `GetStreamInfo` |

The facade lives in [`native/ios/Facade`](../../native/ios/Facade) and is the file to change when
you need something it does not expose yet.

## Namespaces

```csharp
using AntMedia.WebRTC.Api;   // Android: IWebRTCClient, WebRTCClientBuilder, IWebRTCListener
using Org.Webrtc;            // Android: SurfaceViewRenderer, VideoTrack, PeerConnection …
using AntMedia.Net.iOS;      // iOS: AMSClient, AMSClientDelegate, AMSMode
```

The Android Java package is `io.antmedia.webrtcandroidframework`, which the binding generator would
map to `IO.Antmedia.Webrtcandroidframework` — a mouthful that reads nothing like the package you
installed. It is mapped to `AntMedia.WebRTC` instead, sub-namespaces following (`.Api`, `.Core`,
`.Websocket`).

`Org.Webrtc` is deliberately left as-is: it is what the WebRTC project calls itself everywhere, so
it is the name to search for, and every Android WebRTC example online spells it that way.

## Known gaps

**iOS: members typed in Google WebRTC terms are not bound.** `getStats`, `onStats`,
`registerStatsListener`, `setDegradationPreference`, `deliverExternalPixelBuffer`,
`getLocalVideoTrack` and `getLocalAudioTrack` are all absent, because binding them would force
`AntMedia.Net.iOS` to also bind libwebrtc's entire Objective-C surface. `trackAdded` and
`trackRemoved` are surfaced as a track id and kind (`"audio"`/`"video"`) rather than
`RTCMediaStreamTrack`. Video rendering is unaffected — `SetLocalView`/`SetRemoteView` take a
`UIView`.

**Android: one listener member is renamed.** `IWebRTCListener.onNewVideoTrack` is overloaded in
Java, and both overloads generate the same event handler name, which does not compile. The
two-argument overload binds as **`OnNewVideoTrackWithId(VideoTrack, string)`**; the one-argument
overload keeps `OnNewVideoTrack(VideoTrack)`.

**Mac Catalyst is not supported.** Neither of Ant Media's iOS xcframeworks ships a Catalyst slice —
both carry `ios-arm64` and `ios-arm64_x86_64-simulator` only — and their `Package.swift` declares
`.iOS(.v12)`. Supporting it would mean rebuilding libwebrtc for the Catalyst destination.

**No live-server verification.** CI runs offline smoke tests on an Android emulator and an iOS
simulator: they prove the native libraries load, the bound types resolve and calls dispatch. They
do not prove that streaming works, which needs a running Ant Media Server.

## Deprecated package IDs

These were published as `1.0.0.1-beta1` placeholders and are **not** part of this release. Nothing
depends on them and they will not be updated:

- `AntMedia.Net.Mac`
- `AntMedia.Net.WebRTC.Dependency`, `.Android`, `.iOS`, `.Mac`

The iOS WebRTC dependency is no longer a separate package — `AntMedia.Net.iOS` ships Ant Media's
`WebRTC.xcframework` itself, wired into consuming projects by a `.targets` file in the package.
