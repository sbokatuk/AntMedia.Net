## What's changed

**Mac Catalyst streams.** In `2.17.2.2` the Catalyst target frameworks existed only so a MAUI app
could reference the packages without `NU1202` failing the restore — creating a client threw
`PlatformNotSupportedException`. It no longer does. `AntMedia.Net` and `AntMedia.Net.Maui` present
the same client on macOS that they do on Android and iOS, and the sample app publishes and plays
there with no per-platform code.

That works through a new package, **`AntMedia.Net.Mac`**, which `AntMedia.Net` depends on for
`maccatalyst` target frameworks the way it depends on `AntMedia.Net.iOS` for `ios` ones.

The native SDK pins are unchanged: Android is still [WebRTC-Android-SDK][android-sdk] `v2.17.2`
and iOS still [WebRTC-iOS-SDK][ios-sdk] commit [`47e2b02`][ios-commit]. **iOS and Android keep
using Ant Media's own libraries and are untouched by this release**, apart from the packaging
change below.

## Packages

| Package | Platforms | Change |
| --- | --- | --- |
| `AntMedia.Net.Maui` | Android, iOS, Mac Catalyst | Catalyst now resolves a real client |
| `AntMedia.Net` | Android, iOS, Mac Catalyst | depends on `AntMedia.Net.Mac` for `maccatalyst` |
| `AntMedia.Net.Android` | Android | unchanged |
| `AntMedia.Net.iOS` | iOS | 53.5 MB → 40.1 MB, see below |
| `AntMedia.Net.Mac` | Mac Catalyst, Apple Silicon | **new** |

`AntMedia.Net.Mac` previously existed on nuget.org as a `1.0.0.1-beta1` placeholder with no usable
binding in it. This is the first real version of that id.

## How Mac Catalyst is built

The same Ant Media commit, through the same `@objc` facade as iOS, from one `ApiDefinition` linked
into both binding projects — so the two cannot drift, and the managed API is identical.

What differs is the libwebrtc underneath. Ant Media links a **customised** one: their
`WebRTC.framework` exports `RTCAudioDeviceModule`, stock builds do not, and they publish theirs for
iOS only without the fork's source — so no Catalyst slice can be produced from it. The Mac build
therefore links a stock community libwebrtc that does ship `maccatalyst`, and compiles a small shim
beside the facade to supply the API Ant Media's Swift code calls. Their source is not patched.

Two consequences, both worth knowing before you adopt this:

- **Apple Silicon only.** Ant Media selects its renderer with `#if arch(arm64)` — Metal on arm64,
  OpenGL otherwise — and Mac Catalyst has no OpenGL, so there is no `maccatalyst-x64` slice to
  ship. A Catalyst build defaults to both architectures; set
  `<RuntimeIdentifiers>maccatalyst-arm64</RuntimeIdentifiers>` or the link fails on undefined
  `RTCEAGLVideoView`.
- **External audio injection is unavailable on Mac.** That is the one thing
  `RTCAudioDeviceModule` does that the shim cannot stand in for; it logs and drops the sample.
  Camera capture, microphone capture, publish and play all work.

A sandboxed Catalyst app also needs `com.apple.security.device.camera` and
`com.apple.security.device.audio-input` in its entitlements, alongside the usual
`NSCameraUsageDescription` and `NSMicrophoneUsageDescription`. Without them publishing succeeds at
every signalling step and sends no media, which is a confusing way to find out — see
`Platforms/MacCatalyst` in the sample.

## The Apple packages got smaller

`AntMedia.Net.iOS` shipped its xcframeworks twice: once as package content under `native/`, wired
up by a `build/AntMedia.Net.iOS.targets`, and once beside each target framework's assembly. Only
the second copy ever did anything. NuGet's default `PrivateAssets` hides `build` assets from
*transitive* consumers, so an app arriving through `AntMedia.Net.Maui` → `AntMedia.Net` never
imported that `.targets` file at all.

Both `.targets` files and the duplicate payload are gone. Nothing changes for consumers except the
download size — the frameworks still reach the app the same way they always did, through the
binding assembly's `NoBindingEmbedding` payload.

## Upgrading

Nothing to change in your code. If you multi-target `maccatalyst` and were catching
`PlatformNotSupportedException` to skip Mac, you can drop that; if you were excluding the Catalyst
target framework entirely, you can put it back.

Everything in the [2.17.2 notes](2.17.2.md) still applies — the bound API, the Android listener
rename, and what the iOS facade deliberately leaves out.
